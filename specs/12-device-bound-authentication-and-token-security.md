# 12. Device-Bound Authentication and Token Security

## Objective
Replace the current device-identifier-only login model with a stronger authentication design that does not require a third-party identity provider. The new design must require proof that the client still possesses a device-owned private key, must issue short-lived access tokens, must rotate refresh tokens safely, and must protect the authentication surface with rate limiting.

## Problem statement
The current server model treats knowledge of a device identifier as sufficient to obtain a bearer token. That is acceptable for low-risk local development, but it is not strong enough for profile ownership, released-level access, score submission, or future online progression. Shipping a shared client secret inside the app would not materially improve security because a distributed mobile or desktop client cannot keep a shared embedded secret confidential.

## Security goals
- Possession of a device identifier alone must not be enough to authenticate.
- The client must prove possession of a device-owned private key before the server issues new access tokens.
- Access tokens must be short-lived and safe to validate with normal JWT bearer middleware.
- Refresh tokens must be one-time-use rotating secrets with replay detection.
- The server must be able to revoke a specific device or a specific refresh-token family.
- Authentication endpoints must be rate-limited per source and per device identifier.
- The design must work without a full external identity provider.

## Non-goals
- Do not add email/password login, social login, or a hosted identity provider in this slice.
- Do not depend on hardware attestation for the first implementation.
- Do not attempt to make a fully compromised rooted or jailbroken device trustworthy.
- Do not require online access for gameplay that can continue from local cached data.

## Core design

### Device bootstrap
- On first launch, the client generates:
  - a stable `deviceId`
  - an asymmetric public/private key pair intended only for FloodRush authentication
- The private key must remain on the device and must be stored using platform-secure facilities.
- The public key may be sent to the server during registration.
- FloodRush must not ship a shared client secret in the client binary.

### Device registration
- Add `POST /api/profiles/device/register`.
- The request must include:
  - `deviceId`
  - `publicKey`
  - `keyAlgorithm`
  - optional client metadata such as app version and platform
- Registration behavior:
  - If the device record does not exist, the server creates the profile and stores the device credential.
  - If the device record already exists with the same public key, the operation is idempotent.
  - If the device record already exists with a different public key, the server rejects the request unless an explicit future key-rotation flow is used.
- The response must include at least:
  - `profileId`
  - `deviceId`
  - registration status
  - public-key fingerprint or equivalent diagnostic identifier

### Challenge-response login
- Replace the current direct token issuance flow with a two-step login:

#### 1. Create login challenge
- Add `POST /api/profiles/device/challenge`.
- The request includes `deviceId`.
- The server returns:
  - `challengeId`
  - `nonce`
  - `expiresAtUtc`
  - the expected signing algorithm identifier
- Challenges must be:
  - random and unguessable
  - short-lived
  - single-use
  - bound to the requesting device identifier

#### 2. Complete login
- Add `POST /api/profiles/device/login`.
- The request includes:
  - `deviceId`
  - `challengeId`
  - `signature`
  - any required signing metadata
- The client signs a canonical payload containing the server nonce and identifiers using the device private key.
- The server verifies:
  - the device exists and is active
  - the challenge exists, is unexpired, and has not already been used
  - the signature matches the registered public key
- On success, the server issues:
  - a short-lived JWT access token
  - a rotating opaque refresh token
  - token expiry metadata
  - device and profile identifiers needed by the client

## Token model

### Access token
- Access tokens remain JWT bearer tokens.
- Access tokens must include, at minimum:
  - `device_id`
  - `profile_id`
  - `jti`
  - issuer and audience metadata
- Access tokens must be short-lived. The default lifetime should be **15 minutes** and remain configurable.
- The server must validate issuer, audience, signature, and expiry on every protected request.

### Refresh token
- Refresh tokens must be opaque random secrets generated by the server.
- Refresh tokens must never be JWTs.
- Refresh tokens must be stored hashed on the server rather than in plaintext.
- Refresh tokens must be tied to:
  - the device
  - the profile
  - a token family or session family identifier
  - creation and expiry timestamps
  - replacement and revocation metadata
- The server may return the plaintext refresh token only in the response that creates it.

## Refresh-token rotation

### Refresh endpoint
- Add `POST /api/profiles/device/refresh`.
- The request includes the current refresh token.
- The response returns:
  - a new access token
  - a new refresh token
  - new expiry metadata

### Rotation rules
- Every successful refresh invalidates the presented refresh token immediately.
- Every successful refresh returns a brand-new refresh token.
- The client must replace its locally stored refresh token atomically after a successful response.
- Refresh tokens must support:
  - a sliding inactivity window
  - an absolute maximum lifetime
- Default behavior:
  - inactivity timeout: **7 days**
  - absolute maximum lifetime: **30 days**
- These values must be configurable through server options.

### Reuse detection
- If the server receives a refresh token that was already rotated away, consumed, or revoked, the server must treat that as a potential replay attack.
- On detected reuse, the server must revoke the entire token family for that device session.
- After family revocation, the client must perform a fresh challenge-response login and must not continue silently.

## Revocation
- The server must support revocation of:
  - an individual device credential
  - an active refresh-token family
- Revoked devices must not receive new login challenges, access tokens, or refreshed sessions.
- Revocation metadata must be auditable with timestamps and reason codes where practical.

## Rate limiting

### General requirements
- Authentication endpoints must use ASP.NET Core rate limiting with explicit policies.
- Policies must key on source IP address and, when present, normalized `deviceId`.
- The server must return HTTP `429 Too Many Requests` with `Retry-After` when a limit is exceeded.
- Throttled requests must emit structured logs and metrics so abuse and tuning are observable.

### Default policies
- `POST /api/profiles/device/register`
  - per IP: **5 requests per minute**
  - per device: **3 requests per 10 minutes**
- `POST /api/profiles/device/challenge`
  - per IP: **10 requests per 5 minutes**
  - per device: **5 requests per minute**
- `POST /api/profiles/device/login`
  - per IP: **10 requests per 5 minutes**
  - per device: **5 requests per minute**
- `POST /api/profiles/device/refresh`
  - per IP: **60 requests per 5 minutes**
  - per device: **30 requests per 5 minutes**
- These defaults must be configurable.

## Persistence model
- Keep the implementation inside `src\Server\HexMaster.FloodRush.Server.Profiles`.
- Persist the following server-owned records with stable explicit keys:
  - registered device credentials
  - login challenges
  - refresh-token family/session state
  - revoked device or session records
- Azure Table Storage remains the default persistence model for local development.
- Short-lived challenge records may be cleaned up by age-based background maintenance or opportunistic deletion.

## Client requirements
- The MAUI client must generate and hold the private key locally.
- The private key must be stored using platform-secure storage APIs.
- The client must store the refresh token in secure storage.
- The client must automatically:
  - register the device if needed
  - request and sign login challenges
  - use the access token for API requests
  - refresh tokens when the access token is near expiry
- If refresh fails because the token is expired, revoked, or replay-detected, the client must clear the affected auth session and perform a full login flow before retrying protected server work.
- Offline-first behavior remains unchanged: cached levels, local progress, and settings must continue to work while disconnected.

## API contract guidance
- New DTOs must live under `src\Shared\HexMaster.FloodRush.Shared.Contracts`.
- Use explicit versionable contract types for:
  - device registration request and response
  - login challenge request and response
  - login completion request and response
  - token refresh request and response
- Do not expose persistence entities directly.
- Do not include private-key material in any contract.

## Validation and security rules
- Require HTTPS for all authentication and refresh endpoints.
- Reject malformed device identifiers, public keys, signatures, challenge identifiers, and refresh tokens.
- Normalize device identifiers consistently before rate limiting, lookup, and persistence.
- Challenges must expire quickly; the default lifetime should be **5 minutes** and remain configurable.
- Challenges must become unusable immediately after a successful login.
- Refresh tokens and access tokens must never be written to application logs.
- Error responses should avoid leaking internal signing or storage details.

## Migration from the current flow
- The existing device-identifier-only login path is deprecated.
- The target production design is:
  - device registration
  - challenge-response login
  - rotating refresh tokens
- A temporary migration path may exist only to bind a public key to already-known device records during rollout.
- If a temporary migration path is used, it must be:
  - time-boxed
  - explicitly documented
  - removable without changing the long-term contract design

## Testing expectations
- Add unit tests for:
  - device identifier normalization and validation
  - registration idempotency
  - challenge expiry and single-use enforcement
  - signature verification
  - refresh-token rotation
  - refresh-token reuse detection
  - revocation handling
  - rate-limit policy behavior
- Add integration tests for:
  - new-device registration and login
  - successful token refresh
  - expired challenge rejection
  - revoked-device rejection
  - refresh-token replay causing family revocation
  - `429` responses and retry metadata

## Acceptance criteria
- A client can authenticate without a third-party identity provider by proving possession of a device private key registered with the server.
- Knowing only a `deviceId` is no longer sufficient to obtain a bearer token.
- The server issues short-lived JWT access tokens and rotating opaque refresh tokens.
- Refresh-token replay is detected and causes token-family revocation.
- Auth endpoints are protected by configurable rate limits and return `429` with `Retry-After` when exceeded.
- Revoking a device prevents future login and refresh operations for that device.
- Offline-first gameplay remains available when server authentication is unavailable.
